La propuesta técnica se fundamenta en el diseño de un agente conversacional inteligente especializado en asesoramiento financiero personal. El sistema combina procesamiento de lenguaje natural con análisis financiero estructurado, implementando una arquitectura dual que permite comparar un agente LangGraph frente a una implementación clásica basada en clasificación de intenciones.

\subsection{Arquitectura General del Sistema}
\label{subsec:arquitectura}

El sistema sigue una arquitectura modular con cuatro capas principales, como se ilustra en la Figura \ref{fig:arquitectura_capas}.

La capa de presentación proporciona una interfaz web interactiva con métricas financieras en tiempo real, gráficos de distribución de gasto, panel de objetivos activos y un toggle que permite cambiar entre el modo agente y el modo clásico sin reiniciar la sesión.

La capa de agentes contiene dos implementaciones paralelas cuyos detalles se describen en las subsecciones siguientes. La capa de analytics implementa el motor de análisis financiero con trece funciones especializadas reutilizables por ambas implementaciones. La capa de datos gestiona la carga y procesamiento de transacciones mediante pandas, y la persistencia de memoria de usuario en archivos JSON individuales.

El componente central implementa un grafo dirigido de seis nodos conectados mediante aristas fijas y condicionales. La Figura \ref{fig:grafo_langgraph} muestra su estructura completa.

\newpage
\subsubsection{Estado del grafo}

El estado compartido entre nodos se define mediante un \texttt{TypedDict} que agrupa tres campos: la lista de mensajes de conversación (con un reducer acumulativo \texttt{add\_messages}), la memoria persistente del usuario (objetivos, alertas emitidas y resumen acumulado) y las alertas generadas en la ejecución actual. El reducer \texttt{add\_messages} de LangGraph asigna identificadores únicos a cada mensaje y gestiona su eliminación mediante \texttt{RemoveMessage}, lo que es fundamental para el nodo de resumen.

\subsection{Sistema de Memoria Persistente}
\label{subsec:memoria_persistente}

\subsubsection{Estructura de datos}

La memoria de cada usuario se serializa en un archivo JSON individual en el directorio \texttt{memory/}. La estructura incluye cuatro campos: \texttt{goals} (lista de objetivos activos con categoría, límite, fecha límite y fecha de creación), \texttt{past\_alerts} (claves de alertas ya emitidas para evitar repeticiones), \texttt{savings\_tips\_given} (consejos previos, reservado para uso futuro) y \texttt{summary} (resumen acumulado de conversaciones anteriores).

\newpage
\subsubsection{Sistema de alertas}

El nodo \texttt{check\_goals} evalúa todos los objetivos activos al final de cada interacción. Las alertas se generan cuando el porcentaje consumido supera el 80\% o el 100\% del límite, y se registran en \texttt{past\_alerts} con una clave compuesta \texttt{\{categoria\}\_\{YYYY-MM\}\_\{umbral\}} para evitar repeticiones. El umbral se redondea a decenas, de modo que si el porcentaje pasa del 80\% al 95\% en el mismo mes, se generan dos alertas independientes.

Las alertas se inyectan automáticamente al final de la respuesta del agente, garantizando que el usuario siempre esté informado independientemente de qué haya preguntado.

\subsection{Nodo de Resumen Automático}
\label{subsec:nodo_resumen}

\subsubsection{Motivación}

Los modelos de lenguaje tienen una ventana de contexto finita (128k tokens en Llama 3.3 70B vía Groq). En conversaciones largas, el historial acumulado en \texttt{state[``messages'']} puede exceder este límite, provocando errores en la API o degradación de calidad. La estrategia de resumen automático soluciona este problema de forma transparente para el usuario.

\subsubsection{Funcionamiento}

El nodo \texttt{summarize} se ejecuta en cada turno de conversación, entre \texttt{load\_memory} y \texttt{agent}. Si el número de mensajes en el estado no supera el umbral (\texttt{SUMMARIZATION\_THRESHOLD = 20}), el nodo retorna sin modificar el estado. Si lo supera, toma todos los mensajes excepto los 6 más recientes, los convierte a texto plano y los envía a un LLM resumidor dedicado (con temperatura baja y \texttt{max\_tokens} limitado). Si ya existía un resumen previo, el prompt solicita integrarlo con los nuevos mensajes. Tras obtener el nuevo resumen, este se almacena en \texttt{memory[``summary'']} y los mensajes resumidos se eliminan del estado mediante \texttt{RemoveMessage}.

\subsubsection{Parámetros configurables}

Los parámetros del resumen se centralizan en \texttt{config.py}: \texttt{SUMMARIZATION\_THRESHOLD} (20 mensajes antes de activar el resumen), \texttt{KEEP\_RECENT\_MESSAGES} (6 mensajes recientes que se preservan literalmente) y \texttt{SUMMARY\_MAX\_TOKENS} (400 tokens máximos para el resumen generado).

\subsection{El Coach Clásico como Línea Base}
\label{subsec:coach_clasico}

La implementación clásica (\texttt{FinancialCoach}) sirve como línea base para evaluar cualitativamente las mejoras introducidas por el agente LangGraph. Su flujo es determinista y de una sola pasada:

\begin{enumerate}
    \item El mensaje del usuario se clasifica mediante un score de keywords en cuatro intenciones: \texttt{savings\_advice}, \texttt{concept\_explanation}, \texttt{spending\_query} y \texttt{general\_chat}.
    \item Según la intención, un bloque \texttt{if/elif} en \texttt{prompts.py} determina qué funciones de analytics llamar y ensambla un bloque de contexto fijo.
    \item El contexto completo (perfil + datos + historial reciente) se envía al LLM en una única llamada.
    \item La respuesta se devuelve directamente sin inspección del resultado.
\end{enumerate}

La ventana deslizante de historial se limita a \texttt{MAX\_HISTORY\_TURNS = 10} turnos (20 mensajes) sin gestión de contexto adicional.

\subsection{Interfaz de Usuario}
\label{subsec:interfaz}

La interfaz Streamlit (\texttt{app.py}) ofrece un diseño de dos columnas. El sidebar izquierdo presenta las métricas financieras del perfil (ingresos totales, gastos totales, tasa de ahorro), el gráfico de distribución de gastos por categoría, los objetivos activos del usuario (solo en modo agente), el toggle de modo agente/clásico y el selector de modelo LLM. El área principal contiene el historial de conversación y el campo de entrada de texto.

El toggle entre modos es especialmente relevante para la evaluación: permite al usuario comparar en tiempo real las respuestas de ambas implementaciones ante la misma pregunta. Al cambiar de modo, la instancia del coach se recrea y la conversación se reinicia, manteniendo los datos de memoria en disco intactos.

\subsection{Implementación Técnica}
\label{subsec:implementacion_tecnica}

El proyecto se implementa con Python 3.12. Las dependencias principales son LangGraph \cite{langgraph2024} para el grafo del agente, LangChain-Groq para la integración con la API de inferencia \cite{groq2024}, Streamlit para la interfaz \cite{streamlit2021} y pandas para el análisis de datos \cite{pandas2020}. La suite de tests consta de 60 tests distribuidos en cinco módulos:

\begin{itemize}
    \item \texttt{test\_loader.py}: 8 tests --- carga y parseo del CSV de transacciones.
    \item \texttt{test\_analytics.py}: 17 tests --- funciones del motor de analytics.
    \item \texttt{test\_agent\_tools.py}: 18 tests --- 8 de herramientas de analytics y 8 de gestión de objetivos.
    \item \texttt{test\_memory.py}: 9 tests --- carga, guardado, roundtrip y compatibilidad con memorias antiguas.
    \item \texttt{test\_summarize.py}: 8 tests --- estructura del grafo, configuración y comportamiento del nodo de resumen (con LLM mockeado).
\end{itemize}

Todos los tests se ejecutan en menos de 4 segundos sin conexión a la API, gracias al uso de mocks para los componentes LLM.
